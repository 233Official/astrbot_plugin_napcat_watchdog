# NapCat QQ 存活监控插件产品需求文档

- 文档状态：已对齐，待实现
- 文档日期：2026-07-21
- 目标读者：公开维护者、贡献者与部署者
- 适用项目：`astrbot_plugin_napcat_watchdog`

---

## 1. 背景与目标

部分 NapCat QQ 实例仅用于养号，不接入 AstrBot 对话适配器，但维护者仍需要及时获知这些实例是否与监控服务保持连接。

本插件在 AstrBot 内自建轻量 `aiohttp` WebSocket 服务端。多个 NapCat OneBot 11 WebSocket Client 使用同一公网 WSS 地址和全局 Bearer Token 主动连接，插件通过 `X-Self-ID` 自动登记 QQ，并将上线和掉线事件通知到由管理员在群聊内订阅的 AstrBot 群会话。

MVP 目标如下：

- 监控 1–20 个 NapCat QQ 实例。
- 无需预配置 QQ 白名单即可自动登记合法连接。
- 通过群聊管理员命令完成订阅、取消订阅和状态查询。
- 对短时断连和短时离线进行 90 秒防抖。
- 持久化状态、订阅和待补发摘要，避免重启重复通知。
- 使用公开、可部署的 Caddy WSS/TLS 与 NapCat 配置示例。

## 2. 产品流程

### 2.1 服务启动

- 插件启动 `aiohttp` WebSocket 服务端。
- 服务默认在 AstrBot 容器内监听 `0.0.0.0:19090`，默认路径为 `/napcat-watchdog/ws`。
- 公网入口由同一 Docker 网络中的 Caddy 提供 WSS/TLS，并通过容器服务名反向代理到插件监听地址；宿主机不得直接向公网映射 `19090`。
- 管理员在 WebUI 中配置对 NapCat 可见的 `public_ws_url`。

### 2.2 Token 初始化与轮换

- 插件使用一个全局 `access_token`，所有 NapCat 共用该 Token。
- 当 `access_token` 为空时，插件首次启动必须使用密码学安全随机源生成至少 32 字节随机数据形成 Token，并写回 AstrBot 插件配置。
- 自动生成后，管理员可在 WebUI 中手动修改 Token。
- Token 修改后，插件必须关闭全部现有 NapCat 连接；客户端只有使用新 Token 重新鉴权后才能恢复连接。
- Token 原值及其前缀、后缀、哈希不得出现在日志、命令输出、通知或状态持久化文件中。

### 2.3 NapCat 接入与自动登记

- 所有 NapCat 使用相同的 `public_ws_url` 和全局 Bearer Token 建立 OneBot 11 WebSocket Client 连接。
- 插件从合法连接请求的 `X-Self-ID` 自动识别并登记 QQ。
- 不配置实例白名单，也不为单个 QQ 配置独立 Token。
- `X-Self-ID` 缺失或格式非法时拒绝连接。
- Bearer Token 缺失或错误时拒绝连接。
- 同一 QQ 建立新的合法连接时，新连接替换旧连接，不产生掉线或上线通知。

### 2.4 群订阅

- AstrBot 管理员在 QQ 群内执行 `/napcat_watchdog subscribe`，使当前群订阅全部当前和未来自动登记的 NapCat 状态。
- 插件保存当前群的完整 UMO，以及执行订阅命令的管理员 QQ，后者作为该群的通知负责人。
- 同一群重复订阅必须幂等，不创建重复记录，并将负责人更新为最近一次执行订阅的管理员 QQ。
- 订阅成功后回复确认消息和当前 NapCat 状态摘要。
- 新订阅不补发订阅前的历史上线、掉线或通知失败事件。
- 管理员在群内执行 `/napcat_watchdog unsubscribe` 后，取消当前群订阅，并停止向该群发送或补发通知。

### 2.5 状态查询

- AstrBot 管理员执行 `/napcat_watchdog status` 查看连接、NapCat 状态和群订阅摘要。
- 状态查询不得修改连接、状态或订阅。
- 输出不得包含 Token、内部监听地址以外的敏感网络信息或完整持久化数据。

## 3. 总体架构

```text
多个 NapCat OneBot 11 WebSocket Client
                  |
                  | 同一 WSS 地址 + 全局 Bearer Token
                  v
             Caddy WSS/TLS
                  |
                  | WS 反向代理
                  v
      插件 aiohttp WebSocket 服务端
                  |
                  +-- X-Self-ID 自动登记
                  +-- 状态机与 JSON 持久化
                  +-- 已订阅群 UMO 通知
```

架构约束如下：

- 插件仅消费 OneBot 11 lifecycle 与 heartbeat 事件。
- 普通消息、notice、request 和其他非目标事件必须丢弃，不进入状态机或持久化。
- 插件不得向 NapCat 下发任何 OneBot API。
- 所有订阅群接收全部当前和未来 NapCat 的状态事件，不支持按 QQ 筛选。
- 状态处理和通知发送不得阻塞 AstrBot 主事件循环。

## 4. 兼容基线

首版开发与验收基线如下：

- AstrBot 4.26.x，当前参考版本为 4.26.5。
- Python 3.10 及以上版本。
- NapCat 当前稳定 4.x。
- NapCat 必须支持 OneBot 11 WebSocket Client、lifecycle、heartbeat、`X-Self-ID` 和 Bearer Token。
- 不满足上述版本或能力要求的旧版本不属于 MVP 验收范围。

## 5. WebUI 最小配置

WebUI 仅提供以下配置：

- `listen_host`：容器内监听地址，默认 `0.0.0.0`，用于允许同一 Docker 网络中的 Caddy 访问。
- `listen_port`：本地监听端口，默认 `19090`。
- `ws_path`：WebSocket 路径，默认 `/napcat-watchdog/ws`。
- `public_ws_url`：NapCat 使用的公网 WSS 完整地址。
- `access_token`：全局 Bearer Token，使用标准字符串配置字段；为空时由插件首次启动自动生成。
- `offline_timeout_seconds`：掉线防抖时间，默认 `90` 秒。

配置要求如下：

- `public_ws_url` 应使用 `wss://`，并与 Caddy 公网入口及 `ws_path` 一致。
- `access_token` 仅允许 AstrBot 管理员在插件配置编辑页面查看和修改；MVP 不承诺标准配置组件具备密码掩码能力。
- 修改监听配置后，插件应按 AstrBot 插件生命周期安全重建监听服务。
- 修改 `access_token` 后，必须关闭现有连接并要求全部客户端重新鉴权。
- MVP 规模为 1–20 个同时受监控的 QQ。
- WebUI 不配置实例、目标群、负责人或通知平台账号。

## 6. 鉴权与连接管理

### 6.1 全局鉴权

- WebSocket 握手必须携带正确的全局 Bearer Token。
- Token 比较失败、Token 缺失或 Token 尚不可用时拒绝连接。
- 拒绝连接只记录限频安全日志，不发送群通知。
- 安全日志不得记录 Token、Token 前后缀、Token 哈希或完整敏感请求信息。

### 6.2 身份与自动登记

- 合法连接必须包含格式有效的 `X-Self-ID`。
- `X-Self-ID` 作为 NapCat QQ 的唯一业务标识。
- 首次见到合法 `X-Self-ID` 时，插件自动创建该 QQ 的状态记录。
- 已登记 QQ 后续重连应复用原状态记录，不创建重复实例。
- 自动登记数量达到 20 个后，应拒绝新的未知 `X-Self-ID`，但允许已登记 QQ 重连。
- 并发首次连接必须原子占用登记名额，任何时刻累计登记数不得穿透 20 个上限。

### 6.3 连接替换

- 同一 QQ 同时只保留一个当前连接。
- 新合法连接建立后，先将其设为当前连接，再安全关闭旧连接。
- 因合法连接替换而关闭旧连接不得启动掉线计时，也不得发送掉线或上线通知。

---

## 7. 事件消费

- lifecycle 可用于确认连接生命周期和必要的连接元信息。
- heartbeat 仅在 `post_type == "meta_event"`、`meta_event_type == "heartbeat"` 且
  `status.online` 为布尔值时有效；事件包含 `self_id` 时必须与握手的 `X-Self-ID` 一致。
- 有效 heartbeat 用于刷新最后有效心跳时间和读取 `online` 状态。
- 缺少必要字段或格式非法的 lifecycle、heartbeat 不得刷新状态时间。
- 普通消息、notice、request 及其他事件立即丢弃。
- 被丢弃事件不产生通知、状态迁移或事件持久化，也不逐条输出常规日志。
- 插件不发送主动 OneBot API 请求。

## 8. 状态模型

### 8.1 对外状态

每个自动登记的 QQ 对外只需呈现以下状态：

- 在线：存在有效连接，且未达到掉线条件。
- 待确认掉线：已出现断连、心跳缺失或 `online=false`，但尚未持续到防抖阈值。
- 掉线：异常持续达到 `offline_timeout_seconds`，默认 90 秒。

群通知只使用“上线”和“NapCat 掉线”，不区分 QQ 离线、WS 中断或心跳缺失的具体原因。

### 8.2 掉线条件与防抖

以下任一信号可启动同一套掉线计时：

- 当前 WebSocket 连接断开。
- 自最后一次有效 heartbeat 起未再收到有效 heartbeat。
- 有效 heartbeat 持续报告 `online=false`。

默认判定规则如下：

- WebSocket 断开后持续 90 秒未恢复，确认“NapCat 掉线”。
- 自最后一次有效 heartbeat 起达到 90 秒仍无新有效心跳，确认“NapCat 掉线”。
- 新连接建立后尚未收到 heartbeat 时，以连接建立时刻作为临时心跳基准。
- 从首次连续 `online=false` 的有效 heartbeat 起持续达到 90 秒，确认“NapCat 掉线”。
- 上述异常在 90 秒内恢复时，取消待确认状态，不发送掉线或上线通知。
- 多种异常信号同时存在时只维护一个掉线过程，不重复计时或通知。
- 多种异常信号先后出现时采用最早异常起点，后续信号不得推迟掉线截止时间。
- 同一次持续掉线只发送一次掉线通知。
- 运行时期限和持续时间使用 monotonic clock 计算；持久化和展示时间使用 UTC epoch。

### 8.3 上线与恢复

- 首次见到某个 `X-Self-ID` 的合法连接时，立即登记为在线，并向当时已有的全部订阅群发送一次上线通知。
- 若没有订阅群，仍登记在线状态，但不保存供未来补发的上线消息。
- 已知且持久化为在线的 QQ 因插件重启重新连接时，不发送上线通知。
- 已确认掉线的 QQ 建立合法连接后，立即恢复在线并发送一次上线通知。
- 已确认掉线的 QQ 在现有连接上重新出现有效恢复信号时，也立即恢复在线并发送一次上线通知。
- 同一次恢复只发送一次上线通知。
- 同一 QQ 的合法连接替换不视为首次上线或恢复。

### 8.4 重启语义

- 插件启动时加载已登记 QQ 及其最近状态。
- 已知且持久化为在线的 QQ 在插件启动后获得新的 90 秒运行时重连宽限期；宽限期不改写持久化状态。
- 重启不得把已知在线 QQ 的重新连接当作首次上线。
- 重启不得把持续掉线重复通知为新的掉线事件。
- 重启后发生真实状态变化时，按正常防抖与恢复规则处理。
- 插件停机时长不计入运行时 90 秒防抖期限。

## 9. 群订阅与管理员命令

### 9.1 命令列表

- `/napcat_watchdog subscribe`：订阅当前和未来全部 NapCat 状态。
- `/napcat_watchdog unsubscribe`：取消当前群订阅。
- `/napcat_watchdog status`：查看连接、状态和订阅摘要。

### 9.2 权限与上下文

- 全部命令仅 AstrBot 管理员可用。
- `subscribe` 和 `unsubscribe` 仅可在群聊中执行。
- 私聊执行 `subscribe` 或 `unsubscribe` 时必须拒绝，并提示应在目标群中操作。
- `status` 可在 AstrBot 支持的命令上下文中执行，但不得向非管理员泄露摘要。

### 9.3 订阅记录

每个订阅群至少保存：

- 完整 UMO，用于精确恢复通知目标。
- 执行最近一次订阅的管理员 QQ，作为负责人。
- 订阅创建时间和最近更新时间。

行为要求如下：

- 同一群的订阅以 UMO 所表达的会话目标唯一识别。
- 重复订阅不新增记录，只更新负责人和更新时间。
- 订阅成功回复确认信息，以及当前已登记 QQ 的状态摘要。
- 当前无已登记 QQ 时，摘要应明确显示暂无实例。
- 新订阅只接收订阅成功后的事件，不补发历史事件。
- 取消订阅后删除或停用该群记录，并放弃该群尚未成功发送的待补发摘要。

### 9.4 状态摘要

`status` 至少展示：

- WebSocket 服务是否正在监听。
- 当前连接数和已登记 QQ 数。
- 各 QQ 的在线、待确认掉线或掉线状态，以及最后有效心跳时间。
- 当前订阅群数量及脱敏后的订阅摘要。
- 待补发通知数量。

## 10. 通知规则

### 10.1 通知范围

- 所有订阅群接收全部 NapCat 的后续上线和掉线事件。
- 不支持按 QQ 订阅。
- 不通过 WebUI 配置目标群或 `notification_platform_id`。
- 插件使用订阅时保存的完整 UMO 发送群通知。

### 10.2 消息内容与提及

掉线消息至少包含：

- QQ。
- 状态“NapCat 掉线”。
- 确认掉线时间。
- 对该群保存的订阅管理员 QQ 的提及。

上线消息至少包含：

- QQ。
- 状态“上线”。
- 上线或恢复时间。
- 恢复场景可包含离线持续时长。

通知约束如下：

- 掉线消息必须提及该群当前记录的订阅负责人。
- 首次上线和恢复上线消息均不得提及负责人。
- 群消息不区分 QQ 离线、WS 中断或心跳缺失原因。
- 不发送周期重复提醒。
- 消息不得包含 Token、内部地址、客户端地址或其他敏感连接信息。

### 10.3 多群发送与失败处理

- 每个订阅群独立发送并记录结果。
- 单个群发送失败不得阻止其他群发送。
- 单个通知发送失败后最多重试 3 次，不得无限重试。
- 超过重试上限仍失败时，持久化该群的最小待补发摘要。
- 待补发摘要仅保存事件标识、QQ、状态、事件时间、目标 UMO、负责人、发送状态和生成消息所需的最小字段。
- 插件重启后继续处理仍有效订阅的待补发摘要。
- 补发成功后删除或标记完成，避免重复发送。
- 群取消订阅后，不再发送对应待补发通知。
- 没有订阅群时仍维护 NapCat 状态，但不创建、保存或在未来补发历史群消息。

---

## 11. 持久化

状态、订阅和待补发摘要必须保存到 AstrBot 分配的插件数据目录。

### 11.1 存储方式

- 使用轻量 JSON 文件，不引入 SQLite 或其他数据库。
- 每次更新先写入同目录临时文件，成功后原子替换正式文件。
- 持久化文件不得写入源码目录或提交到公开仓库。
- 文件损坏或版本不兼容时记录脱敏错误并 fail closed：保留或隔离原文件，拒绝启动状态服务和
  WebSocket 接入，不得静默按空状态继续运行。

### 11.2 最小持久化内容

- 已自动登记的 QQ。
- 每个 QQ 的最近确认状态和状态变更时间。
- 当前掉线过程、掉线通知和恢复通知的去重标记。
- 已确认掉线的开始时间，用于计算恢复时长。
- 订阅群完整 UMO、负责人 QQ、创建时间和更新时间。
- 各群最小待补发通知摘要及发送状态。
- 连接对象、connection generation、monotonic 时间、运行时 deadline 和最后 heartbeat 不持久化。
- 首次登记及确认状态迁移时立即原子写盘；普通 heartbeat 不逐次写盘。

### 11.3 隐私边界

- `access_token` 由 AstrBot 插件配置保存，不写入状态 JSON。
- 状态 JSON 不得保存 Token 原值、前后缀或哈希。
- UMO 和负责人 QQ 仅用于订阅恢复与通知，不得输出到无权限日志或公开诊断信息。

---

## 12. 安全与可观测性

- 使用密码学安全随机源生成全局 Token，随机数据不少于 32 字节。
- Token 仅在 AstrBot 管理员可访问的插件配置编辑页面中保存和修改；自定义密码掩码页面不属于 MVP。
- Token 轮换后立即使旧连接失效。
- 日志、通知、命令输出、异常信息和状态 JSON 均不得泄露 Token 或其派生标识。
- 非法连接仅记录脱敏且限频的安全日志，不触发群通知。
- 被丢弃的 OneBot 事件不逐条输出常规日志。
- 插件应记录启动、停止、监听状态、合法连接、连接替换、自动登记、状态迁移、订阅变更、通知结果和持久化错误。
- 插件停止或重载时必须释放监听服务、连接、定时器和后台任务。

## 13. Caddy 配置示例

以下示例使用保留域名，将公网 WSS 入口反向代理到插件默认监听地址：

```caddyfile
watchdog.example.com {
    reverse_proxy /napcat-watchdog/ws astrbot:19090
}
```

对应 WebUI 配置示例：

```text
listen_host = 0.0.0.0
listen_port = 19090
ws_path = /napcat-watchdog/ws
public_ws_url = wss://watchdog.example.com/napcat-watchdog/ws
```

部署要求如下：

- 插件监听端口仅通过 Docker 内部网络提供给 Caddy，不直接映射到宿主机公网。
- Caddy 负责公网 DNS、证书和 TLS 终止。
- 修改 `ws_path` 时，Caddy 路由、`public_ws_url` 和 NapCat 地址必须同步修改。
- 示例不包含真实域名、服务器地址或凭据。

## 14. NapCat 配置示例

每个 NapCat 使用相同的公网 WSS 地址和全局 Token。以下为结构示例，实际字段名以部署时使用的 NapCat 4.x 版本为准：

```json
{
  "name": "watchdog-client",
  "enable": true,
  "url": "wss://watchdog.example.com/napcat-watchdog/ws",
  "token": "${NAPCAT_WATCHDOG_ACCESS_TOKEN}",
  "reconnectInterval": 5000,
  "heartInterval": 30000
}
```

配置要求如下：

- `url` 使用 WebUI 中的 `public_ws_url`。
- `token` 使用插件全局 `access_token`，并作为 Bearer Token 发送。
- 所有 NapCat 使用同一 Token，但各自通过 `X-Self-ID` 自动登记。
- heartbeat 建议每 30 秒发送一次，且必须包含可用的 `online` 状态。
- 重连策略应能在默认 90 秒防抖窗口内尝试恢复。
- 公开配置、日志和问题报告必须替换实际域名、QQ 和 Token。

## 16. 验收标准

### 16.1 Token

- `access_token` 为空时，首次启动自动生成基于至少 32 字节安全随机数据的 Token，并写回插件配置。
- 后续启动复用已有 Token，不重复生成。
- 管理员修改 Token 后，全部现有连接被关闭，旧 Token 无法重新连接，新 Token 可以重新鉴权。
- WebUI、日志、通知、命令输出和状态 JSON 均不展示 Token 原值、前缀、后缀或哈希。

### 16.2 自动登记与连接

- 使用正确全局 Bearer Token 和合法 `X-Self-ID` 时连接成功并自动登记 QQ。
- 无需实例白名单或每实例 Token 即可登记 1–20 个 QQ。
- Token 错误、Token 缺失、`X-Self-ID` 缺失或格式非法时连接被拒绝。
- 第 21 个未知 QQ 被拒绝，已登记 QQ 仍可重连。
- 并发未知 QQ 连接不会使累计登记数超过 20。
- 同一 QQ 的新合法连接替换旧连接，且不产生上线或掉线通知。
- 旧连接在被替换后产生的残留事件或断连回调不会影响当前连接状态。
- 普通消息、notice、request 被丢弃，插件不下发 OneBot API。

### 16.3 命令权限与订阅

- 仅 AstrBot 管理员可执行 `subscribe`、`unsubscribe` 和 `status`。
- `subscribe`、`unsubscribe` 在私聊中被拒绝，在群聊中可正常执行。
- 订阅保存完整 UMO 和执行订阅的管理员 QQ。
- 同一群重复订阅不产生重复记录，并更新负责人。
- 订阅成功回复确认和当前状态摘要，不补发历史事件。
- `unsubscribe` 停止当前群后续通知和待补发。
- 重启后 UMO、负责人和订阅状态正确恢复。

### 16.4 上线、掉线与恢复

- 首次合法连接自动登记后，向当时已有订阅群发送一次不带提及的上线通知。
- 已知在线 QQ 因插件重启重新连接时不发送上线通知。
- 已知在线 QQ 在插件重启后获得新的 90 秒重连宽限期，插件停机时长不计入掉线判断。
- WS 断开持续不足 90 秒并恢复时不通知，达到 90 秒时只发送一次掉线通知。
- 最后有效 heartbeat 后不足 90 秒恢复时不通知，达到 90 秒无有效心跳时只发送一次掉线通知。
- `heartbeat.online=false` 持续不足 90 秒并恢复时不通知，达到 90 秒时只发送一次掉线通知。
- 已确认掉线后恢复时立即发送一次不带提及的上线通知，可包含离线时长。
- 多种掉线信号并存时，同一次掉线只通知一次；恢复只通知一次。
- 群消息统一显示“NapCat 掉线”，不区分具体故障原因。

### 16.5 多群通知与提及

- 所有订阅群接收全部 NapCat 的后续状态事件。
- 每个群的掉线消息提及该群当前负责人，上线消息不提及任何用户。
- 多群分别记录发送结果，单群失败不阻止其他群。
- 每条失败通知最多重试 3 次，超过上限后保存最小待补发摘要。
- 重启后继续补发仍有效订阅的通知，成功后不再重复。
- 没有订阅群时仍维护状态，但不保存或在未来补发历史群消息。

### 16.6 JSON 持久化

- 状态、订阅和待补发摘要保存到 AstrBot 插件数据目录的轻量 JSON 文件。
- 每次更新使用同目录临时文件写入后原子替换。
- 重启后恢复自动登记状态、订阅 UMO、负责人、掉线去重标记和待补发摘要。
- 重启不会重复发送已完成的掉线或上线通知。
- 损坏或版本不兼容的状态文件会阻止状态服务和 WebSocket 接入，不会静默清空历史登记。
- heartbeat 不逐次写盘，运行时 connection generation、monotonic deadline 和连接对象不进入 JSON。
- 未引入 SQLite 或其他数据库。

### 16.7 配置与部署文档

- WebUI 仅包含约定的六个最小配置字段及正确默认值。
- `public_ws_url`、Caddy 路径和 NapCat WS Client 地址保持一致。
- 文档包含可修改的 Caddy WSS/TLS 反向代理示例。
- 文档包含使用全局 Bearer Token、`X-Self-ID` 和 heartbeat 的 NapCat OneBot 11 WebSocket Client 示例。
- 示例仅使用保留域名与占位凭据，不包含真实 QQ、群号、服务器或 Token。

### 16.8 兼容与规模

- AstrBot 4.26.x（参考 4.26.5）和 Python 3.10 及以上通过测试。
- NapCat 当前稳定 4.x 的 OneBot 11 WebSocket Client、lifecycle、heartbeat、`X-Self-ID` 和 Bearer Token 通过集成验收。
- 1–20 个 QQ 同时监控场景通过验收。
