# NapCat QQ 存活监控插件产品需求文档

- 文档状态：已对齐，待实现
- 文档日期：2026-07-21
- 目标读者：公开维护者、贡献者与部署者
- 适用项目：`astrbot_plugin_napcat_watchdog`

---

## 1. 背景与目标

部分 NapCat QQ 实例仅用于养号，不接入 AstrBot 的对话适配器，但维护者仍需要及时获知 QQ 离线、连接中断或实例从未成功接入监控等异常。

本插件应在 AstrBot 内提供一个轻量、只读的监控入口：由各 NapCat 实例通过 OneBot 11 WebSocket Client 主动连接插件自建的 `aiohttp` WebSocket 服务，插件根据连接与心跳事件维护状态，并通过指定的 AstrBot 通知账号向多个目标群发送异常及恢复通知。

产品目标如下：

- 监控 1–20 个未接入 AstrBot 对话适配器的 NapCat 养号 QQ 实例。
- 对 QQ 离线、连接丢失和从未连接进行稳定、可解释、可恢复的状态判定。
- 对告警、防抖、恢复和重启连续性提供明确且可测试的行为。
- 对 Token、网络地址和日志实施最小暴露与严格脱敏。
- 通过 AstrBot WebUI 完成实例和通知配置，通过管理员命令查询只读状态。

---

## 2. 非目标

本阶段不包含以下能力：

- 使用 WebUI 轮询 NapCat 状态。
- 对主机、容器或进程进行精确诊断。
- 区分 NapCat 进程退出、主机故障与网络故障。
- 实现完整 OneBot 11 适配器。
- 转发普通消息、通知事件或请求事件。
- 向 NapCat 下发主动 OneBot API 调用。
- 提供维护静默窗口。
- 对同一持续异常进行周期重复告警。
- 提供高可用或集群部署能力。

---

## 3. 用户与使用场景

### 3.1 目标用户

- 管理多个 NapCat 养号实例的 AstrBot 管理员。
- 负责实现、测试和维护本插件的公开仓库贡献者。
- 需要依据公开文档部署 Caddy、AstrBot 与 NapCat 的运维人员。

### 3.2 核心场景

- 管理员在 WebUI 中预配置允许接入的 NapCat 实例及其独立 Token。
- NapCat 启动后主动连接公网 WSS 入口，由 Caddy 终止 TLS 并反向代理至插件本地 WS 服务。
- 插件接收 lifecycle 与 heartbeat 事件，持续维护各实例状态。
- 实例异常时，插件通过唯一指定的 AstrBot 通知账号向多个目标群发送一次告警。
- 实例恢复时，插件向相同目标群发送一次恢复通知。
- 管理员使用 `/napcat_watchdog status` 查询所有实例的只读状态。

## 4. 总体架构

数据流如下：

```text
NapCat OneBot 11 WebSocket Client
              |
              | WSS
              v
        Caddy 公网入口
              |
              | WS 反向代理
              v
AstrBot 插件 aiohttp WebSocket 服务
              |
              +-- 状态机与持久化
              |
              +-- AstrBot 通知账号 -> 多个目标群
```

架构约束如下：

- 每个 NapCat 必须使用 OneBot 11 WebSocket Client 主动连接插件。
- 公网入口必须由 Caddy 提供 WSS/TLS；插件自身不负责公网 TLS 终止。
- 插件 WS 服务默认监听 `127.0.0.1:19090`，默认路径为 `/napcat-watchdog/ws`。
- 监听地址、端口和路径均可配置。
- 插件仅消费 lifecycle 与 heartbeat 事件。
- 普通消息、notice、request 及其他非目标事件必须被丢弃，不进入业务处理或持久化。
- 插件不得向 NapCat 下发任何 OneBot API 请求。

### 4.1 MVP 兼容基线

首版开发与验收以以下版本和能力为基线：

- AstrBot 4.26.x，当前参考版本为 4.26.5。
- Python 3.10 及以上版本。
- NapCat 当前稳定 4.x，且必须支持 OneBot 11 WebSocket Client、lifecycle、heartbeat、`X-Self-ID` 和 Bearer Token。
- 不满足上述版本或能力要求的旧版本不属于 MVP 验收范围。

## 5. 配置需求

所有业务配置由 AstrBot WebUI 管理。敏感字段应使用密码输入控件，持久化到 AstrBot 本地插件配置中，不得以明文回显。

### 5.1 WebSocket 服务配置

- `listen_host`：监听地址，默认 `127.0.0.1`。
- `listen_port`：监听端口，默认 `19090`。
- `ws_path`：WebSocket 路径，默认 `/napcat-watchdog/ws`。

### 5.2 实例白名单

WebUI 应支持配置 1–20 个实例。每个实例包含：

- `name`：公开展示用实例名，便于告警和状态查询识别。
- `qq_id`：该实例的 QQ 标识，必须与连接请求中的 `X-Self-ID` 完全匹配。
- `token`：该实例独立使用的随机 Token，由用户生成。
- `enabled`：是否启用该实例的接入和监控。

实例配置要求如下：

- 不同实例不得共用 Token。
- `qq_id` 在启用实例中必须唯一。
- `token` 必须以密码字段保存并严格脱敏。
- 禁用实例不得通过鉴权，也不得产生在线状态、异常告警或恢复通知。
- 配置中不得预置真实 QQ、群号、服务器地址或 Token。

### 5.3 通知配置

- `notification_platform_id`：唯一的 AstrBot 通知账号标识，只允许配置一个。
- `target_group_ids`：接收通知的目标群列表，支持多个目标群。
- `mention_user_ids`：异常告警中需要提及的用户列表，可为空。

通知规则如下：

- 所有目标群均由同一个 `notification_platform_id` 账号发送通知。
- 异常告警可提及 `mention_user_ids`。
- 恢复通知不得提及任何用户。
- 目标群和提及用户的配置不得在公开示例中使用真实标识。

---

## 6. 连接鉴权与身份校验

### 6.1 鉴权要求

- 每个连接必须通过 Bearer Token 携带与预配置实例匹配的 Token。
- Token 校验必须使用该实例自己的 Token，不能使用全局共享 Token。
- Token 缺失、错误、对应实例被禁用或无法匹配白名单时，连接必须被拒绝。
- Token 不得出现在正常日志、错误日志、状态命令、通知消息或持久化事件摘要中。

### 6.2 身份校验

- 插件必须读取连接请求中的 `X-Self-ID`。
- `X-Self-ID` 必须与通过 Token 匹配到的实例 `qq_id` 完全一致。
- `X-Self-ID` 缺失、格式非法或与 `qq_id` 不一致时，连接必须被拒绝。
- 已通过鉴权的同一 QQ 建立新连接时，新连接替换旧连接；插件主动关闭旧连接，但不得因此产生异常或恢复通知。

### 6.3 非法连接日志

- 非法连接只记录安全日志，不触发业务告警。
- 安全日志必须限频，默认按“同一来源 + 同一拒绝原因”统计，每分钟最多记录 5 条。
- 超过默认限额的同类日志应聚合计数或暂时静默，避免扫描、错误配置或重试风暴造成日志刷屏。
- 日志可以记录脱敏后的拒绝原因、时间和必要计数，不得记录 Token、Token 前缀、Token 后缀、Token 哈希、完整公网地址或其他敏感连接信息。

---

## 7. 事件消费与心跳

- 插件仅处理 OneBot 11 lifecycle 与 heartbeat 事件。
- heartbeat 的标准间隔为 30 秒。
- 连续 90 秒未收到有效 heartbeat 时，视为心跳超时并进入连接中断判定流程。
- lifecycle 可用于记录连接生命周期，但不得替代 heartbeat 对 QQ 在线状态的判定。
- heartbeat 中的 `online` 字段用于 QQ 在线状态判定。
- 普通消息、notice、request 和其他事件必须立即丢弃，不产生通知、状态迁移或事件持久化。
- 格式错误或缺少必要字段的 lifecycle、heartbeat 事件不得刷新最后有效心跳时间。

---

## 8. 状态模型与判定规则

### 8.1 对外异常分类

- `QQ_OFFLINE`（QQ 已离线）：连接仍可用，但连续收到 3 个 `online=false` 的有效 heartbeat。
- `CONNECTION_LOST`（监控连接已丢失）：已成功连接过的实例发生连接关闭或心跳超时，并超过 90 秒重连窗口仍未恢复有效连接。
- `NEVER_CONNECTED`（启动后从未连接）：启用实例在插件启动或实例被启用后，经过 180 秒启动宽限期仍未建立过合法连接。

正常状态不作为异常分类。实现可维护内部过渡状态，但对外展示和通知必须归并到上述分类或正常状态。

### 8.2 启动宽限期

- 插件启动时，每个已启用实例获得 180 秒宽限期。
- 实例从禁用切换为启用时，重新获得 180 秒宽限期。
- 宽限期内未连接不得告警。
- 宽限期结束仍从未建立合法连接时，触发一次 `NEVER_CONNECTED` 告警。
- 宽限期内首次建立合法连接时，取消该实例的从未连接等待状态。

### 8.3 QQ 离线防抖

- 只有连续 3 个有效 heartbeat 均为 `online=false` 时，才进入 `QQ_OFFLINE`。
- 任意一个 `online=true` 会立即清空尚未达到阈值的离线计数。
- 已处于 `QQ_OFFLINE` 时，首个有效 `online=true` 必须立即恢复正常并发送恢复通知。
- 重复的 `online=false` 不得对同一次持续异常重复告警。

### 8.4 连接丢失与重连窗口

- 以最后有效 heartbeat 为时间起点，连续 90 秒未收到新的有效 heartbeat 后，实例进入内部的待重连状态。
- 进入待重连状态后再保留 90 秒重连窗口；因此，从最后有效 heartbeat 到触发 `CONNECTION_LOST` 告警总计约 180 秒。
- 合法连接显式断开时可提前标记连接不可用，但不得跳过上述基于最后有效 heartbeat 的告警时间线。
- 在窗口内恢复合法连接并收到有效 heartbeat 时，不产生 `CONNECTION_LOST` 告警。
- 超过重连窗口仍未恢复时，触发一次 `CONNECTION_LOST` 告警。
- `CONNECTION_LOST` 后恢复合法连接并收到有效 heartbeat 时，立即恢复正常并发送一次恢复通知。
- 同一 QQ 的新合法连接替换旧连接不属于连接丢失，不启动重连窗口，也不发送通知。

### 8.5 异常切换

- 每个实例任一时刻只对外呈现一个当前异常类型。
- 当前异常类型发生变化时，应结束旧异常并开启新异常，分别记录事件；通知不得把两个异常合并为无法判定的通用故障。
- 同一异常持续期间只发送一次异常告警。
- 同一异常恢复时只发送一次恢复通知。
- 不进行周期重复提醒。

---

## 9. 通知需求

### 9.1 发送范围

- 使用唯一配置的 `notification_platform_id` 发送通知。
- 每次通知发送到全部 `target_group_ids`。
- 异常告警可提及 `mention_user_ids`；恢复通知不提及用户。

### 9.2 消息内容

异常和恢复消息至少包含：

- 实例名。
- QQ 标识。
- 异常类型及清晰的中文说明。
- 异常发生时间。
- 最后有效心跳时间；从未收到心跳时应明确显示“无”。
- 截至发送时的持续时长。

消息不得包含：

- Token 或其可逆片段。
- 内部域名、服务器地址、客户端地址或反向代理地址。
- 与故障定位无关的连接参数。

### 9.3 通知失败处理

- 单个通知发送失败后最多重试 3 次，重试必须是有限的，不得无限循环。
- 多个目标群应分别记录发送结果，单个目标群失败不得阻止其他目标群发送。
- 超过重试上限仍失败时，持久化最小待补发摘要。
- 待补发摘要只保存重新发送所需的实例标识、事件类型、事件时间、目标群标识、发送状态和必要消息字段，不保存 Token 或敏感地址。
- 插件重启后应继续处理待补发摘要。
- 补发成功后必须标记完成或删除对应待补发记录，避免重复发送。

---

## 10. 持久化与重启连续性

状态与事件必须保存到 AstrBot 分配的插件数据目录，不得写入源码目录或公开仓库文件。

首版持久化采用轻量 JSON 文件，不引入 SQLite 或其他数据库。每次更新必须先写入同目录临时文件，在写入成功后通过原子替换更新正式文件，避免进程中断造成部分写入。

至少持久化以下信息：

- 实例是否曾成功建立合法连接。
- 当前状态或异常类型。
- 当前异常的开始时间。
- 最后有效心跳时间和最后已知 `online` 状态。
- 当前离线防抖计数所需的最小状态。
- 异常告警是否已发送。
- 恢复通知是否已发送。
- 最近必要的状态迁移事件摘要。
- 未成功发送的待补发通知摘要及各目标群发送状态。

重启行为要求如下：

- 插件必须读取已有持久化状态，避免对重启前已告警且仍持续的同一异常重复告警。
- 重启不得把持续异常错误地标记为恢复。
- 重启后的连接判断仍遵循 180 秒启动宽限期，但不得抹除“曾连接”和既有异常通知记录。
- 状态文件损坏或版本不兼容时，应记录脱敏错误并采用安全降级策略，不得泄露配置中的敏感字段。
- JSON 持久化更新应使用“临时文件写入成功后原子替换”方式保证崩溃安全性，避免部分写入导致整个状态不可读取。

## 11. 管理命令与 WebUI

### 11.1 管理员命令

- 仅提供只读命令 `/napcat_watchdog status`。
- 仅 AstrBot 管理员可以执行该命令。
- 非管理员调用时必须拒绝访问，不返回实例状态详情。
- 命令输出应包含每个启用实例的名称、脱敏 QQ、当前状态、最后心跳和异常持续时长。
- 命令输出不得包含 Token、内部地址、客户端地址或完整连接参数。
- 命令不得执行重连、重启、配置修改、告警确认或其他写操作。

### 11.2 WebUI

- WebUI 用于管理 WS 服务、实例白名单和通知配置。
- Token 字段必须使用密码输入控件并在 UI 中掩码展示，不得在状态页、列表页、日志等非编辑上下文中回显原值。
- WebUI 不承担状态轮询或实时监控职责。
- 配置保存时必须校验实例数量、必填字段、`qq_id` 唯一性和 Token 独立性。

---

## 12. 安全与隐私要求

- 插件遵循白名单接入，未配置实例一律拒绝。
- 每个实例使用独立随机 Token，并由部署者负责安全生成与分发。
- Token 只保存在 AstrBot 本地配置和 NapCat 本地连接配置中。
- 所有日志、命令输出、通知、异常栈附加信息和持久化事件摘要均不得包含 Token。
- Token 即使需要辅助识别也只能显示固定占位符，不得展示前缀、后缀或哈希值。
- 公开文档和示例只能使用保留示例域名、占位 QQ、占位群号和环境变量式 Token 占位符。
- 公网连接使用 WSS/TLS；插件默认仅监听回环地址。
- 非法连接日志必须按“同一来源 + 同一拒绝原因每分钟最多 5 条”的默认规则限频，超过后聚合或静默，且不得由非法连接触发群告警。

---

## 13. 可观测性与运行约束

- 插件应记录启动、停止、合法连接建立、连接替换、状态迁移、通知结果和持久化失败等必要日志。
- 日志应以实例名和脱敏 QQ 作为主要关联信息。
- 被丢弃的普通消息、notice、request 不应逐条输出常规日志。
- 非法连接和格式错误事件应使用限频日志。
- 所有定时器、连接和后台任务必须在插件卸载或停止时正确释放。
- 在 1–20 个实例规模内，心跳处理不得阻塞 AstrBot 主事件循环。

## 14. Caddy 配置示例

以下示例仅展示公开 WSS 入口到本地插件 WS 服务的反向代理关系，部署者必须替换示例域名并按实际环境管理证书：

```caddyfile
watchdog.example.com {
    reverse_proxy /napcat-watchdog/ws 127.0.0.1:19090
}
```

部署要求如下：

- 公网客户端连接地址为 `wss://watchdog.example.com/napcat-watchdog/ws`。
- 插件保持默认 `127.0.0.1:19090` 监听时，不应直接暴露该端口到公网。
- 修改插件 `ws_path` 时，Caddy 路由与 NapCat WebSocket Client 地址必须同步修改。
- Caddy 配置、DNS、证书与防火墙管理属于部署职责，不由插件自动修改。

## 15. NapCat 配置示例

每个 NapCat 实例应单独创建 OneBot 11 WebSocket Client，并使用该实例对应的 QQ 标识与独立 Token。以下为结构示例，具体字段名以部署时使用的 NapCat 版本为准：

```json
{
  "name": "watchdog-client",
  "enable": true,
  "url": "wss://watchdog.example.com/napcat-watchdog/ws",
  "token": "${NAPCAT_WATCHDOG_TOKEN}",
  "reconnectInterval": 5000,
  "heartInterval": 30000
}
```

配置要求如下：

- `url` 指向 Caddy 提供的 WSS 地址，不直接指向插件回环监听地址。
- `token` 使用该 QQ 在 WebUI 白名单中配置的独立随机 Token。
- heartbeat 间隔设置为 30 秒。
- 断开后的重连间隔或重连策略应确保可在 90 秒窗口内尝试恢复；部署者应依据 NapCat 实际配置能力调整。
- NapCat 发起连接时必须携带正确的 `X-Self-ID`，并与白名单 `qq_id` 一致。
- 公开配置、问题报告和日志中必须将 Token、QQ 标识及实际域名脱敏。

## 16. 验收标准

### 16.1 鉴权与身份校验

- 使用正确实例 Token 且 `X-Self-ID` 匹配 `qq_id` 时，连接成功。
- Token 缺失、错误、复用其他实例 Token 或实例被禁用时，连接被拒绝。
- `X-Self-ID` 缺失或与 Token 对应实例的 `qq_id` 不一致时，连接被拒绝。
- 同一 QQ 建立第二个合法连接时，新连接替换旧连接，且不产生异常或恢复通知。
- 非法连接不会触发群告警；高频非法连接只产生限频且无敏感信息的安全日志。

### 16.2 事件消费与状态机

- lifecycle 与 heartbeat 可以进入对应处理流程。
- 普通消息、notice、request 被丢弃，且不改变实例状态。
- 有效 heartbeat 每 30 秒到达时，最后心跳持续更新。
- 从最后有效 heartbeat 起 90 秒未收到新 heartbeat 后进入待重连流程；再等待 90 秒仍未恢复时，只发送一次 `CONNECTION_LOST` 告警，即总计约 180 秒后告警。
- 启动或启用后 180 秒内从未连接不告警；超过 180 秒仍未连接时只发送一次 `NEVER_CONNECTED` 告警。

### 16.3 QQ 离线防抖与恢复

- 连续 1 次或 2 次 `online=false` 不触发告警。
- 连续第 3 次 `online=false` 触发一次 `QQ_OFFLINE` 告警。
- 达到阈值前收到 `online=true` 会清零离线计数。
- `QQ_OFFLINE` 后首个 `online=true` 立即触发一次恢复通知。
- 持续 `online=false` 不产生周期重复告警。

### 16.4 通知

- 告警由唯一配置的 `notification_platform_id` 发送到全部 `target_group_ids`。
- 异常告警按配置提及 `mention_user_ids`，恢复通知不提及用户。
- 消息包含实例名、QQ、异常类型、发生时间、最后心跳和持续时长。
- 消息和日志不包含 Token、Token 前缀、Token 后缀、Token 哈希、内部地址或客户端地址。
- 同一异常只告警一次，同一恢复只通知一次。

### 16.5 持久化与通知失败

- 异常产生后重启插件，持续中的同一异常不会重复告警。
- 恢复事件产生后重启插件，不会重复发送已完成的恢复通知。
- 通知发送失败时最多重试 3 次，不发生无限重试。
- 超过重试上限后存在脱敏的待补发摘要；重启后可继续补发。
- 多目标群中一个目标发送失败时，其他目标仍可成功发送并独立记录结果。
- 状态和事件文件位于 AstrBot 插件数据目录，不写入源码目录。
- 状态与事件使用轻量 JSON 保存，并通过同目录临时文件写入后原子替换；未引入 SQLite 或其他数据库。

### 16.6 权限与配置

- 管理员执行 `/napcat_watchdog status` 可查看全部启用实例的只读状态。
- 非管理员执行命令被拒绝，且看不到实例详情。
- 命令输出不包含 Token、内部地址或完整连接参数。
- WebUI 可配置 1–20 个实例，并拒绝重复 `qq_id`、共享 Token、缺失必填字段或超过规模上限的配置。
- WebUI Token 使用密码字段并以掩码展示，且不在非编辑上下文回显原值。
- 日志、通知、命令输出和持久化摘要均不得展示 Token 前缀、后缀或哈希值。

### 16.7 部署文档

- 文档包含可复制修改的 Caddy WSS 反向代理示例。
- 文档包含 NapCat OneBot 11 WebSocket Client 示例。
- 示例使用保留域名和占位 Token，不包含真实 QQ、群号、服务器或凭据。
- 按文档配置后，NapCat 可通过 Caddy WSS 连接插件默认 WS 路径。

### 16.8 兼容基线

- AstrBot 4.26.x（参考 4.26.5）和 Python 3.10 及以上版本通过开发与验收。
- NapCat 当前稳定 4.x 可通过 OneBot 11 WebSocket Client 发送 lifecycle、heartbeat、`X-Self-ID` 和 Bearer Token，并通过相关集成验收。
